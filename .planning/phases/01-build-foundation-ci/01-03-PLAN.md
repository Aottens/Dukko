Triggers (D-13):
  push: branches: ["**"]
  pull_request: branches: ["**"]

Runner (per CLAUDE.md §"CI / Packaging"):
  runs-on: macos-14

Job 1: build-and-validate (Release)
  Steps:
    1. checkout (with submodules: recursive — JUCE may be a submodule per plan 02 TBD #3)
    2. Cache CPM sources at ~/Library/Caches/CPM, key: hashFiles('**/CMakeLists.txt', '**/cmake/*.cmake')
    3. Configure: cmake -B build -DCMAKE_BUILD_TYPE=Release -DCMAKE_OSX_ARCHITECTURES=arm64
    4. Build: cmake --build build --config Release --parallel
    5. Verify VST3 arm64: lipo -archs build/Dukko_artefacts/Release/VST3/Dukko.vst3/Contents/MacOS/Dukko | grep -w arm64
    6. Verify CLAP arm64: lipo -archs build/Dukko_artefacts/Release/CLAP/Dukko.clap/Contents/MacOS/Dukko | grep -w arm64
    7. Download pluginval 1.0.4: curl pluginval_macOS.zip + unzip + chmod +x
    8. Run pluginval --strictness-level 10 --validate-in-process
    9. Run Steinberg validator: VALIDATOR=$(find build/_deps -name validator -type f | head -1); "$VALIDATOR" Dukko.vst3
    10. Download clap-validator 0.3.2: curl + tar
    11. Run clap-validator validate Dukko.clap
    12. LICENSES.md staleness guard: for d in build/_deps/*/; do n=$(basename "$d" | sed 's/-src$//'); grep -qi "$n" LICENSES.md || { echo "MISSING in LICENSES.md: $n"; exit 1; }; done
    13. Upload VST3 artifact (retention-days: 30)
    14. Upload CLAP artifact (retention-days: 30)

Job 2: build-debug-asan (Debug + ASan compile-only)
  Steps:
    1. checkout (with submodules: recursive)
    2. Cache CPM (same key as job 1)
    3. Configure: cmake -B build-debug -DCMAKE_BUILD_TYPE=Debug -DCMAKE_OSX_ARCHITECTURES=arm64 -DCMAKE_CXX_FLAGS="-fsanitize=address -fno-omit-frame-pointer"
    4. Build: cmake --build build-debug --config Debug --parallel

Note on clap-validator asset name (TBD #7): the 0.3.2 release publishes a macOS arm64 tarball. The workflow uses the **hardcoded direct asset URL** `https://github.com/free-audio/clap-validator/releases/download/0.3.2/clap-validator-0.3.2-macos-arm64.tar.gz` (matches GitHub's standard `releases/download/<tag>/<asset-filename>` URL convention). If this exact URL is wrong on first CI run (asset filename differs), the curl step fails loudly with HTTP 404 — the executor patches the URL in plan 04 as a one-line fix. This is preferred over a dynamic GitHub API call (which is rate-limited unauthenticated, ~60 req/hr per IP, and silently returns empty if the schema changes).

Pamplejuce already ships a build.yml. We are REWRITING it (after plan 01 already renamed job names) to match the D-10/D-11/D-12/D-13/D-14 spec exactly, not patching the existing template.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner ↔ external download URLs | The workflow curls pluginval (Tracktion) and clap-validator (free-audio) binaries from GitHub releases on every run. |
| Forked-PR workflow ↔ repo secrets | Default GitHub Actions behaviour for public repos: workflows triggered by PRs from forks get a read-only GITHUB_TOKEN and no access to repo secrets. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | pluginval / clap-validator binary downloads in CI | accept | Both binaries are pinned to specific release versions (pluginval `v1.0.4`, clap-validator `0.3.2`) in the workflow URLs. They are CI-only tools (per LICENSES.md note from plan 02) and do not link into Dukko. Trust model is "GitHub serves what GitHub serves"; same as plan 01 T-01-01. |
| T-03-02 | Elevation of Privilege | Workflow injection via untrusted PR | mitigate | The workflow uses GitHub Actions defaults: PRs from forks run with read-only `GITHUB_TOKEN` and no access to repo secrets. The workflow itself does not use `pull_request_target` (which would expand the threat surface). No explicit `permissions:` block needed because we are intentionally relying on GitHub's secure defaults for public repos (D-08). |
| T-03-03 | Denial of Service | LICENSES.md staleness guard fails permanently if a future contributor adds a CPM dep without updating LICENSES.md | mitigate (intentional) | This is the desired behaviour — the workflow blocks the merge until LICENSES.md is updated. The `exit 1` with `MISSING in LICENSES.md: <name>` is loud and self-explanatory. Documentation cost: contributors learn the rule on the first failed PR. |
| T-03-04 | Information Disclosure | Workflow logs may expose internal paths under build/_deps/ | accept | Public repo (D-08); paths are not secret. Useful for debugging on the rare CI failure. |
